# Conversation → Article Playbook

把一段有價值的對話，從「當下很有感」轉成「未來可重用的資產」。

## 目的

這份 playbook 是給 Jones + Jarvis 用的標準流程。

核心問題不是「今天聊得深不深」，而是：

> 這段對話能不能被封裝成未來還找得到、看得懂、用得上的東西？

## 適用情境

適合處理這類對話：
- 某個生活觀察引發的深度思考
- 跟 GPT / Jarvis 的來回辯證
- 對某個人生問題、創作問題、產品問題出現了新的框架
- 對話裡已經長出一個可命名的模型或方法論

不適合直接封裝的內容：
- 單純情緒發洩但還沒有形成洞見
- 純資訊查詢、沒有觀點提煉
- 碎片化聊天、沒有明確核心主題

## 標準流程

### Step 1. 判斷是否值得封裝
問這三題：
1. 這段對話有沒有出現一個明確核心問題？
2. 這段對話有沒有長出一個可重述的結論、模型或框架？
3. 三個月後回來看，這段內容還有沒有價值？

如果至少兩題是「有」，就值得封裝。

### Step 2. 保留原始材料
把原始對話或來源先保存：
- 原始 markdown 對話
- Slack/WhatsApp/GPT 對話轉存
- Jarvis 與 Jones 的討論稿

原始材料通常放在：
- 同一個資料夾內當 source/original
- 或先放進 repo 裡，之後再決定是否歸檔

### Step 3. 抽出骨架
至少整理出：
- 這篇在回答什麼問題？
- 為什麼這個問題重要？
- 最後形成了什麼結論？
- 有哪些框架、步驟、模型？

### Step 4. 封裝成可讀文章
最低限度要有：
- 標題
- TL;DR
- 結構化段落
- 可以單獨閱讀，不依賴原始對話上下文

### Step 5. 決定放哪裡
根據內容性質判斷：
- 長期有效的框架 → `principles/`
- 可執行流程 → `playbooks/`
- 主題研究 / 報告 / 專案 → `projects/`
- 重要方向選擇與原因 → `decision-log/`

### Step 6. 補回鏈接
如果封裝內容來自原始對話，應盡量加上：
- 原始來源連結
- 相關 project / principle 互相連結
- README 或 index 的入口

## 封裝品質標準

一篇封裝完成的文章，應該滿足：
- 未來的 Jones 看得懂
- 未來的 Jarvis 接得回去
- 不看原始聊天紀錄，也知道它在說什麼
- 能被引用到其他 repo、文件、產品、課程、決策裡

## Jarvis 的角色

Jarvis 在這個流程裡的責任：
- 幫忙判斷這段對話值不值得封裝
- 幫忙抽骨架、命名、整理成文
- 幫忙判斷該放進 principles / playbooks / projects / decision-log 哪一層
- 在得到 Jones 同意後，直接改 repo、commit、push

## 一句話版本

> 對話不是成品，封裝才是成品。
